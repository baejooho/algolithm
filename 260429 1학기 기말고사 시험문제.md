# 수식의 표기법 정리

## 표기법 종류

| 표기법 | 영문 | 연산자 위치 | 예시 |
|--------|------|------------|------|
| 전위 표기법 | prefix notation | 피연산자 **앞** | `+AB` |
| 중위 표기법 | infix notation | 피연산자 **가운데** | `A+B` |
| 후위 표기법 | postfix notation | 피연산자 **뒤** | `AB+` |

> 우리가 평소에 쓰는 수식은 **중위 표기법**이다.  
> 컴퓨터가 계산하기 쉬운 방식은 **후위 표기법**이다.

---

## 중위 → 전위 변환 방법

```
① 우선순위에 따라 괄호로 묶기
② 각 연산자를 해당 왼쪽 괄호 앞으로 이동
③ 괄호 제거
```

### 예시: `A*B - C/D`

```
① 괄호로 묶기   : ( (A*B) - (C/D) )
② 연산자 이동   : -( *(A B) /(C D) )
③ 괄호 제거     : -*AB/CD
```

---

## 중위 → 후위 변환 방법

```
① 우선순위에 따라 괄호로 묶기
② 각 연산자를 해당 오른쪽 괄호 뒤로 이동
③ 괄호 제거
```

### 예시: `A*B - C/D`

```
① 괄호로 묶기   : ( (A*B) - (C/D) )
② 연산자 이동   : ( (AB)* (CD)/ )-
③ 괄호 제거     : AB*CD/-
```

---

## 핵심 비교 정리

```
중위 표기식  :  A * B - C / D   (우리가 쓰는 방식)
전위 표기식  :  - * A B / C D   (연산자가 앞으로)
후위 표기식  :  A B * C D / -   (연산자가 뒤로)
```

---

## 암기 팁

- **전위(prefix)** = **앞(pre)** 에 연산자
- **중위(infix)** = **사이(in)** 에 연산자
- **후위(postfix)** = **뒤(post)** 에 연산자

> 전위/후위 변환 시 **피연산자 순서(A, B, C, D)는 절대 바뀌지 않는다!**  
> 연산자 위치만 이동한다.




# 큐의 구현 - 원형 큐가 필요한 이유

## 순차 큐의 문제점: 포화 상태 잘못 인식

### 상황 설명

순차 큐(1차원 배열로 구현한 큐)에서 삽입과 삭제를 반복하다 보면 아래와 같은 상황이 발생한다.

```
배열 크기 n = 4 (인덱스 0~3)

삽입/삭제 반복 후 현재 상태:

[0]     [1]     [2]     [3]
 빈칸   빈칸    빈칸    [D]
  ↑                     ↑
front                  rear
```

- `[0]`, `[1]`, `[2]` 는 삭제되어 **비어있는 공간**
- 실제로 데이터는 `[3]` 하나뿐
- 그런데 `rear = n-1 = 3` 이므로 **포화 상태로 잘못 인식**
- 앞에 빈자리가 있어도 더 이상 삽입을 수행하지 않는 문제 발생!

---

### 왜 이런 문제가 생기나?

순차 큐는 삽입할 때 `rear`를 오른쪽으로만 이동시킨다.  
삭제 후 앞 공간이 비어도 `rear`는 되돌아오지 않기 때문에,  
`rear`가 배열 끝(n-1)에 닿으면 무조건 포화 상태로 판단해버린다.

```
❌ 잘못된 인식
rear = n-1  →  "꽉 찼다!" (실제로는 앞에 빈 공간 있음)
```

---

### 해결 방법 1 - 원소를 앞으로 이동

삭제 후 남은 원소들을 배열의 앞부분(인덱스 0부터)으로 당겨오는 방법

```
이동 전:               이동 후:
[0]  [1]  [2]  [3]    [0]  [1]  [2]  [3]
 빈   빈   빈  [D]    [D]  빈    빈    빈
       ↑    ↑           ↑   ↑
     front rear       front rear
```

#### 문제점
- 원소를 이동하는 작업 자체가 **연산이 복잡**하고 **비효율적**
- 데이터가 많을수록 이동 비용이 커짐

---

### 해결 방법 2 - 원형 큐(Circular Queue) 사용 ✅

배열의 끝과 처음을 연결하여 **원형으로 사용**하는 방식  
→ `rear`가 배열 끝(n-1)에 도달하면 다시 인덱스 0으로 돌아옴

```
인덱스 이동 방식:  (rear + 1) % n

예) n = 4 일 때
rear = 3  →  (3+1) % 4 = 0  →  다시 처음으로!
```

이렇게 하면 앞에 빈 공간이 있으면 거기에 계속 삽입할 수 있어서  
포화 상태를 **정확하게** 인식할 수 있다.

---

## 핵심 정리

| 구분 | 순차 큐 | 원형 큐 |
|------|---------|---------|
| 포화 조건 | `rear == n-1` | `(rear+1) % n == front` |
| 공백 조건 | `front == rear` | `front == rear` |
| 삭제 후 빈 공간 | 재사용 불가 | 재사용 가능 |
| 효율성 | 낮음 (이동 필요) | 높음 |

---

## 시험 대비 서술 포인트

> **"순차 큐에서 삽입과 삭제를 반복하면, 앞부분에 빈 자리가 있어도 rear = n-1 이 되는 순간 포화 상태로 잘못 인식하여 더 이상 삽입을 수행하지 않는 문제가 발생한다. 이를 해결하기 위해 원소를 앞으로 이동시키는 방법이 있지만, 연산이 복잡하고 비효율적이다. 따라서 배열의 처음과 끝을 연결한 원형 큐를 사용하면 빈 공간을 재활용할 수 있어 이 문제를 효율적으로 해결할 수 있다."**


# 큐의 응용 - 운영체제의 작업 큐

## 핵심 개념

큐는 **선입선출(FIFO)** 구조이기 때문에, 순서를 지켜야 하는 작업 처리에 사용된다.

---

## 운영체제에서 큐를 사용하는 이유

> 먼저 요청한 작업을 먼저 처리해야 하는 상황 = **선입선출 구조 = 큐**

---

## 1. 프린터 버퍼 큐 (Printer Buffer Queue)

- CPU에서 프린터로 보낸 데이터를 **순서대로(선입선출)** 프린터에서 출력하기 위해 사용
- CPU는 데이터를 빠르게 보내지만, 프린터는 느리게 출력함
- 그 사이에 큐(버퍼)를 두어 **보낸 순서대로** 출력되도록 관리

```
CPU  →  [데이터1 | 데이터2 | 데이터3]  →  프린터
              (프린터 버퍼 큐)
         먼저 보낸 것 먼저 출력 (FIFO)
```

---

## 2. 스케줄링 큐 (Scheduling Queue)

- CPU 사용을 요청한 **프로세스들의 순서를 스케줄링**하기 위해 큐를 사용
- 여러 프로세스가 동시에 CPU를 요청할 경우, 큐에 넣어 순서대로 처리

### 프로세스의 상태 흐름

```
생성 상태 → 준비 큐(Ready Queue) → 실행 상태(CPU) → 종료 상태
                  ↑                        |
                  |       대기 큐           |
                  ←──────────────────────←
                  (I/O 등 대기 후 다시 준비 큐로)
```

| 상태 | 설명 |
|------|------|
| 생성 상태 | 프로세스가 새로 만들어진 상태 |
| 준비 상태 | CPU 할당을 기다리는 상태 → **준비 큐**에서 대기 |
| 실행 상태 | CPU를 할당받아 실제로 실행 중인 상태 |
| 대기 상태 | I/O 등 특정 이벤트를 기다리는 상태 → **대기 큐**에서 대기 |
| 종료 상태 | 실행이 끝난 상태 |

---

## 핵심 비교 정리

| 종류 | 사용 목적 | 큐가 필요한 이유 |
|------|-----------|----------------|
| 프린터 버퍼 큐 | CPU → 프린터 데이터 전송 순서 관리 | 보낸 순서대로 출력해야 하므로 |
| 스케줄링 큐 | 프로세스의 CPU 사용 순서 관리 | 요청 순서대로 CPU를 할당해야 하므로 |

---

## 시험 대비 서술 포인트

> **"운영체제에서 큐는 프린터 버퍼 큐와 스케줄링 큐에 사용된다.  
> 프린터 버퍼 큐는 CPU가 프린터로 보낸 데이터를 선입선출 구조로 순서대로 출력하기 위해 사용한다.  
> 스케줄링 큐는 CPU 사용을 요청한 여러 프로세스들을 준비 큐와 대기 큐로 관리하여 순서대로 CPU를 할당하기 위해 사용한다.  
> 두 경우 모두 먼저 들어온 것을 먼저 처리하는 FIFO(선입선출) 특성을 활용한다."**
