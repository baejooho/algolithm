## 1. 자료구조의 이해

### 개념

> 자료를 **효율적으로 표현하고 저장하고 처리**할 수 있도록 정리하는 것

컴퓨터가 효율적으로 문제를 처리하기 위해서는 문제를 정의하고 분석하여 최적의 프로그램을 작성해야 하며, 이를 위해 **자료구조에 대한 개념과 활용 능력**이 필요하다.

### 분류

자료의 형태에 따라 다음과 같이 분류한다.

| 분류 | 설명 | 예시 |
|------|------|------|
| **단순 구조** | 기본 자료형 | 정수, 실수, 문자, 문자열 |
| **선형 구조** | 자료 간 관계가 1:1 | 순차 리스트, 연결 리스트, 스택, 큐, 데크 |
| **비선형 구조** | 자료 간 관계가 1:다 또는 다:다 | 트리, 그래프 |
| **파일 구조** | 레코드의 집합인 파일에 대한 구조 | 순차 파일, 색인 파일, 직접 파일 |

## 2. 자료의 표현

컴퓨터는 숫자, 문자, 그림, 소리 등 **모든 형식의 자료를 2진수 코드**로 표현하여 저장 및 처리한다.

- **2진수 코드** : 1과 0, On과 Off, 참(True)과 거짓(False)의 조합
- n개의 비트로 **2ⁿ개의 상태** 표현 가능

### 수치 자료의 표현

#### 10진수 표현

**존(Zone) 형식**
- 10진수 한 자리를 1바이트(8비트)로 표현
- 상위 4비트(존 영역) : `1111`로 고정
- 하위 4비트(수치 영역) : 해당 10진수 값의 2진수
- 마지막 자리 존 영역에 부호 표시 (양수: `1100`, 음수: `1101`)

**팩(Pack) 형식**
- 10진수 한 자리를 존 영역 없이 4비트로 표현
- 최하위 4비트에 부호 표시 (양수: `1100`, 음수: `1101`)

#### 2진수 정수 표현

| 방식 | 설명 |
|------|------|
| **부호 절댓값** | 최상위 1비트로 부호 표시 (0: 양수, 1: 음수), 나머지 비트로 절댓값 표현 |
| **1의 보수** | 음수를 n비트의 1의 보수로 표현 (각 비트를 반전) |
| **2의 보수** | 1의 보수에 1을 더한 값으로 음수 표현 (현재 가장 널리 사용) |

> 세 방식 모두 양수 표현은 동일하며, 음수 표현 방식만 다르다.

#### 2진수 실수 표현

- **고정 소수점** : 소수점 위치를 최상위 비트 왼쪽에 고정
- **부동 소수점** : 표현 범위가 넓음, 부호/지수부/가수부로 구성
  - 4바이트(단정도) 및 8바이트(배정도) 형식 사용

---

### 문자 자료의 표현

문자에 대한 이진수 코드를 정의하여 표현한다.

| 코드 | 비트 수 | 특징 |
|------|---------|------|
| **BCD** | 6비트 | 존(2비트) + 2진수(4비트), 10진수 0~9와 영어 대문자, 특수문자 표현 |
| **EBCDIC** | 8비트 | 존(4비트) + 2진수(4비트), 대소문자 및 특수문자 표현 가능 |
| **ASCII** | 7비트 | 존(3비트) + 2진수(4비트), 대소문자 및 특수문자 표현 가능 |
| **유니코드** | 16비트(2바이트) | 세계 여러 나라 언어를 통일된 방법으로 표현하는 국제 표준(ISO/IEC 10646) |

> 코드 발전 순서: BCD → EBCDIC → ASCII → **유니코드** (현재 일반화)

유니코드는 XML, Java, CORBA 3.0, WML 등 인터넷 기반 프로그램에 사용된다.

---

### 논리 자료의 표현

논리값(참/거짓)을 1바이트로 표현하는 방식:

| 방법 | 참(True) | 거짓(False) |
|------|----------|------------|
| 방법 1 | `00000001` (최하위 비트 = 1) | `00000000` |
| 방법 2 | `11111111` (전체 비트 = 1) | `00000000` |
| 방법 3 | 하나 이상의 비트가 1 | `00000000` |

---

### 포인터 자료의 표현

- 메모리의 주소를 표현하기 위한 자료 형식
- 변수의 주소나 메모리 특정 위치의 주소를 저장하고 주소 연산에 사용

---

### 문자열(String) 자료의 표현

여러 문자로 이루어진 그룹을 메모리에 연속적으로 저장하는 자료 형식

부분 문자열 표현 방법:

| 방법 | 설명 |
|------|------|
| **방법 1** | 부분 문자열 사이에 구분자(예: `;`) 사용 |
| **방법 2** | 가장 긴 문자열 길이에 맞춰 고정 길이로 저장 |
| **방법 3** | 연속 저장 후 각 부분 문자열에 대한 포인터 사용 |

## 3. 자료의 추상화

### 추상화 개념

- **추상화** : 복잡한 문제를 단순화시켜 쉽게 해결하기 위한 방법
- **자료 추상화** : 처리할 자료, 연산, 자료형에 대한 추상화 표현

### 자료형 구성 요소

| 구성 요소 | 설명 |
|-----------|------|
| **자료** | 프로그램의 처리 대상이 되는 모든 것 |
| **연산** | 어떤 일을 처리하는 과정 (연산자에 의해 수행) |
| **자료형** | 처리할 자료의 집합 + 수행할 연산자의 집합 |

> 예) 정수 자료형: 자료 = {…, -1, 0, 1, …}, 연산자 = {+, -, ×, ÷, mod}

### 추상 자료형 (ADT, Abstract Data Type)

> 자료와 연산자의 특성을 **논리적으로 추상화**하여 정의한 자료형

| 개념 | 질문 | 의미 |
|------|------|------|
| **추상화** | "무엇(what)인가?" | 논리적으로 정의 |
| **구체화** | "어떻게(how) 할 것인가?" | 실제적으로 표현 |

## 4. 알고리즘의 이해

### 정의

> 문제해결 방법을 추상화하여 **단계적 절차를 논리적으로 기술**해 놓은 명세서

### 알고리즘의 5가지 조건

| 조건 | 설명 |
|------|------|
| **입력 (Input)** | 알고리즘 수행에 필요한 자료가 외부에서 입력으로 제공될 수 있어야 함 |
| **출력 (Output)** | 알고리즘 수행 후 하나 이상의 결과를 출력해야 함 |
| **명확성 (Definiteness)** | 명령어들은 명확하게 명세되어야 함 |
| **유한성 (Finiteness)** | 반드시 종료되어야 함 |
| **효과성 (Effectiveness)** | 모든 명령어들은 기본적이며 실행 가능해야 함 |

## 5. 알고리즘의 표현 방법

### 표현 방법 종류

1. **자연어** : 서술적 표현 방법
2. **순서도 (Flow chart)** : 도식화 표현 방법
3. **프로그래밍 언어** : 구체화 방법
4. **가상코드 (Pseudo-code)** : 추상화 방법

---

### 가상코드 (Pseudo-code)

알고리즘 기술언어(ADL, Algorithm Description Language)를 사용하여 프로그래밍 언어와 유사하게 알고리즘을 표현하는 방법

- 특정 프로그래밍 언어가 아니므로 **직접 실행 불가**
- 특정 언어로의 **변환이 용이**

#### 기본 요소

| 요소 | 설명 |
|------|------|
| **기호** | 변수, 자료형, 프로그램 이름 등 (첫 문자는 반드시 영문자) |
| **자료형** | 정수형, 실수형, 문자형, 논리형, 포인터, 문자열 등 |
| **연산자** | 산술 연산자, 관계 연산자, 논리 연산자 |

#### 제어 구조

| 구조 | 설명 |
|------|------|
| **if 문** | 조건에 따라 실행할 명령문 결정 |
| **중첩 if 문** | 다단계 조건 처리 |
| **case 문** | 여러 조건 중 해당 조건 찾아 수행 (중첩 if 문으로 표현 가능) |
| **for 문** | 정해진 횟수만큼 반복 |
| **while-do 문** | 조건이 참인 동안 반복 (선검사) |
| **do-while 문** | 먼저 실행 후 조건 검사 (후검사) |
| **함수문** | 처리 작업별로 모듈화한 부프로그램 |

## 6. 알고리즘의 성능 분석

### 성능 분석 기준

| 기준 | 설명 |
|------|------|
| **정확성** | 올바른 자료 입력 시 유한한 시간 내에 올바른 결과 출력 여부 |
| **명확성** | 알고리즘이 얼마나 이해하기 쉽고 명확하게 작성되었는가 |
| **수행량** | 알고리즘 특성을 나타내는 중요 연산 분석 |
| **메모리 사용량** | 알고리즘 수행에 필요한 저장 공간 |
| **최적성** | 가장 중요한 기준 |

---

### 성능 분석 방법

#### 공간 복잡도 (Space Complexity)

> 알고리즘 실행 완료까지 필요한 **총 저장 공간의 양**

```
공간 복잡도 = 고정 공간 + 가변 공간
```

#### 시간 복잡도 (Time Complexity)

> 알고리즘 실행 완료까지의 **총 소요 시간**

```
시간 복잡도 = 컴파일 시간 + 실행 시간
```

- **컴파일 시간** : 프로그램마다 거의 고정적
- **실행 시간** : 컴퓨터 성능에 따라 달라지므로 **명령문의 실행 빈도수**로 계산
  - 지정문, 조건문, 반복문 내 제어문, 반환문 → 하나의 단위시간을 갖는 기본 명령문으로 취급

---

### 성능 분석 표기법

#### 빅-오 표기법 O(f(n))

> "Big Oh of f(n)"

- **함수의 상한**을 나타내는 표기법
- 최악의 경우에도 `g(n)`의 수행 시간 안에 완료됨을 보장

**수학적 정의**: 모든 n ≥ n₀에 대하여 `|f(n)| ≤ c|g(n)|`을 만족하는 상수 c, n₀이 존재하면 `f(n) = O(g(n))`

- 실행 빈도수 함수에서 **가장 큰 영향을 주는 항**을 선택, 계수는 생략
- 예) 실행 시간 함수가 `4n + 2`이면 → **O(n)**

#### 빅-오메가 표기법 Ω(f(n))

> "Big Omega of f(n)"

- **함수의 하한**을 나타내는 표기법
- 알고리즘 수행에 **최소한** `g(n)`의 시간이 필요함을 의미

**수학적 정의**: 모든 n ≥ n₀에 대하여 `|f(n)| ≥ c|g(n)|`을 만족하는 상수 c, n₀이 존재하면 `f(n) = Ω(g(n))`

#### 빅-세타 표기법 θ(f(n))

> "Big Theta of f(n)"

- **상한과 하한이 같은 정확한 차수**를 표현하는 표기법
- `f(n) = θ(g(n))`이 되려면 `f(n) = O(g(n))`이면서 동시에 `f(n) = Ω(g(n))`이어야 함

**수학적 정의**: 모든 n ≥ n₀에 대하여 `c₁|g(n)| ≤ f(n) ≤ c₂|g(n)|`을 만족하는 상수 c₁, c₂, n₀이 존재하면 `f(n) = θ(g(n))`

---

### 시간 복잡도 비교

성능이 좋은 순서 (실행 시간이 짧은 순서):

```
O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(n³) < O(2ⁿ)
```

| 표기 | 명칭 |
|------|------|
| O(1) | 상수 시간 |
| O(log n) | 로그 시간 |
| O(n) | 선형 시간 |
| O(n log n) | 선형 로그 시간 |
| O(n²) | 이차 시간 |
| O(n³) | 삼차 시간 |
| O(2ⁿ) | 지수 시간 |
